## 한 줄 요약

친구랑 **시나리오 중복 방지**를 위해 `getLearnedExpressions()`가 호출될 때마다 해당 회원의 `mindSessions` **서브컬렉션 전체**를 읽고 있었습니다. 세션이 쌓일수록 Firestore 읽기 비용·지연이 **선형 증가**하는 구조였고, 이를 **`users/{memberId}` 문서의 요약 배열**로 바꿨습니다.

> **전체 회원 DB 스캔이 아닙니다.** 항상 **한 회원** 범위였지만, 그 회원의 **완료 세션 N개 = N문서 읽기**가 문제였습니다.

**현재 상태:** `main` 반영 완료 ✅ (`e567c5f`) — Closes #4

---

## 누가 겪나? (비개발자용)

| 상황 | 영향 |
|------|------|
| 아이가 친구랑 완료 세션 **적음** (MVP) | 체감 거의 없음 |
| 세션 **수십~수백 개** + 시나리오 자주 생성 | Firestore 읽기·응답이 점점 느려질 수 있음 |
| 조회 실패 시 | 빈 목록 → AI가 **이미 배운 표현**을 다시 줄 수 있음 (기존과 동일) |

**쓰이는 곳:** 「이야기 보러 가기」 시나리오 생성, 주간 리포트 학습 요약 등.

## 비유

**수정 전:** 시나리오 만들 때마다 **출석부 전권**을 펼쳐 「이 아이가 배운 말」을 전부 다시 적음.  
**수정 후:** `users/5` 표지에 붙은 **요약 스티커**(`learnedExpressions`)만 본다. 새 학습 완료 시 스티커에 한 줄 추가.

## 수정 전 코드

```kotlin
fun getLearnedExpressions(memberId: Long): List<String> {
    mindSessionsCollection(memberId)
        .get()   // 해당 회원 mindSessions 전체 fetch
        .documents
        .mapNotNull { it.getString("learnedExpression") }
        .distinct()
}
```

| 완료 세션 수 | 요청 1회당 Firestore read |
|-------------|---------------------------|
| 10 | ~10문서 |
| 50 | ~50문서 |
| 100 | ~100문서 |

## 수정 후 동작

### 1) 조회 `getLearnedExpressions()`

1. `users/{memberId}` 문서 **1회** 읽기 → `learnedExpressions` 배열 반환
2. 필드가 **없으면** (레거시 회원): `mindSessions` **1회만** 전체 스캔 → backfill 저장 → 이후는 1문서만

### 2) 저장 `saveMindSession()`

기존처럼 `mindSessions/{setId}` 저장 +  
`learnedExpression`이 있으면 `FieldValue.arrayUnion`으로 user 문서 배열에 **추가** (중복 없음).

```kotlin
// 조회 — 문서 1회
userDocument(memberId).get() → learnedExpressions

// 저장 시 — arrayUnion
mapOf("learnedExpressions" to FieldValue.arrayUnion(expression))
```
