# 스케줄러 설계: 멱등성과 확장 계획

[[기술적-의사결정]]

---

## 스케줄러 목록

| 스케줄러 | 실행 시각 (KST) | 역할 |
|----------|----------------|------|
| `TodoScheduler` | 매일 00:00 | 전체 회원 오늘 할 일 3개 생성 |
| `WeeklyReportScheduler` | 매주 월 01:00 | 주간 성장 리포트 AI 트리거 |
| `WeeklySafetyScheduler` | 매주 월 01:00 | 주간 안전 신호 AI 트리거 |

모든 스케줄러는 `zone = "Asia/Seoul"` 명시 — 서버 타임존과 무관하게 KST로 실행됩니다.

## 멱등성 설계 (TodoScheduler)

```kotlin
// TodoService.generateDailyTodos
fun generateDailyTodos(member: Member, today: LocalDate) {
    // 이미 오늘 할 일이 있으면 스킵
    if (todoRepository.existsByMemberIdAndAssignedDate(member.id, today)) return
    ...
}

// TodoScheduler — 스킵 여부를 로깅
val beforeExists = todoService.isTodayTodoGenerated(member.id, today)
todoService.generateDailyTodos(member, today)
if (beforeExists) skipCount++ else successCount++
```

스케줄러가 배포 재시작 등으로 중복 실행되거나 동일 날짜에 두 번 호출되어도
이미 생성된 회원은 안전하게 스킵합니다.

## 주간 배치 비동기 콜백 패턴

```
[월 01:00 KST] WeeklyReportScheduler
    ↓ 온보딩 완료 회원 목록 조회
    ↓ 1인당 POST /ai/report/weekly 호출
          ↓ (AI가 비동기로 집계·LLM 처리)
          ↓ POST /api/v1/weekly-report (BE 콜백)
               ↓ WeeklyReportCallbackService.receiveReport()
               ↓ DB 저장 (upsert 패턴)
```

주간 집계·LLM 처리는 수십 초가 걸릴 수 있어 동기 대기가 불가능합니다.
스케줄러가 트리거만 던지고 AI가 비동기로 처리한 뒤 BE에 콜백하는 패턴입니다.

### 콜백의 멱등성 (upsert)

```kotlin
val existing = weeklyReportRepository.findByMemberIdAndWeekStart(member.id, request.weekStart)
if (existing != null) {
    existing.updateSections(request.sections)  // 중복 수신 시 덮어쓰기
} else {
    weeklyReportRepository.save(WeeklyReport(...))
}
```

AI가 네트워크 오류로 콜백을 재전송해도 `UNIQUE (member_id, week_start)` 제약 + upsert 패턴으로
중복 레코드 없이 안전하게 처리됩니다.

## 현재 한계와 Spring Batch 전환 계획

### 현재 방식의 문제

```kotlin
// 현재: 전체 회원을 한 번에 메모리에 로드
val members = memberRepository.findAll()  // OOM 위험
members.forEach { member -> ... }
```

회원 수가 적은 MVP 단계에서는 문제 없습니다.
회원이 증가하면 `findAll()`의 메모리 적재 문제와 순차 처리 시간이 늘어납니다.

### Spring Batch 전환 시 개선 포인트

| 항목 | 현재 | Spring Batch |
|------|------|-------------|
| 데이터 로드 | `findAll()` (전체 메모리) | `ItemReader` 페이징 (chunk 단위) |
| 실패 처리 | 개별 try-catch, 로그만 | 실패한 chunk만 재시도 |
| 실행 이력 | 없음 | `JobRepository`로 자동 관리 |
| 병렬 처리 | 순차 | `Step` 파티셔닝으로 병렬화 가능 |

### 분산 환경에서의 중복 실행 문제

현재는 서버가 1대이므로 스케줄러 중복 실행 위험이 없습니다.
서버가 2대 이상이 되면 모든 인스턴스에서 스케줄러가 동시에 실행됩니다.
멱등성으로 데이터 중복은 막지만 AI 트리거가 N배 발송됩니다.

해결 방법:
- **ShedLock** — DB 락으로 단일 실행 보장 (가장 간단)
- **Spring Batch + JobRepository** — 단일 Job 실행 보장
- **Quartz Cluster** — 분산 스케줄링 전용 솔루션
