## 1. 회원 탈퇴: FK 순서 명시적 제어

### 결정

JPA `cascade = REMOVE` 대신 레포지토리 레벨에서 삭제 순서를 직접 제어합니다.

```kotlin
// WithdrawalService — 자식 테이블부터 순서대로 삭제
chatMessageRepository.deleteAllByMemberId(memberId)   // 1. 메시지 (세션 FK)
chatSessionRepository.deleteAllByMemberId(memberId)   // 2. 세션
todoRepository.deleteAllByMemberId(memberId)           // 3. 할 일
rewardHistoryRepository.deleteAllByMemberId(memberId) // 4. 보상 이력
userStatsRepository.deleteByMemberId(memberId)        // 5. 통계
userMathProgressRepository.deleteAllByMemberId(...)   // 6-7. 학습 진행률
...
memberRepository.delete(member)                        // 마지막: 회원
```

### CASCADE 대신 명시적 삭제를 선택한 이유

| 항목 | `cascade = REMOVE` | 명시적 삭제 |
|------|-------------------|------------|
| 실행 쿼리 | 각 엔티티를 SELECT 후 DELETE (N+1 DELETE) | 단일 `DELETE WHERE member_id = ?` |
| 삭제 순서 | JPA 내부 결정 (디버깅 어려움) | 코드에 명시, 읽기 쉬움 |
| FK 위반 디버깅 | 예외 발생 시 추적 어려움 | 순서 코드에서 즉시 파악 가능 |

회원 1명 탈퇴에 관련 테이블이 13개이므로 CASCADE 방식은 수십~수백 개의 SELECT+DELETE가 발생합니다.
명시적 bulk delete로 테이블당 쿼리 1개로 처리합니다.

## 2. 엔티티 설계 원칙

### data class 금지

```kotlin
// JPA Entity — class만 사용
class Member(
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,
    ...
)
```

`data class`는 `equals()`/`hashCode()`를 모든 필드로 자동 생성합니다.
JPA 프록시 객체는 실제 엔티티와 다른 타입이므로 이 자동 생성 메서드가 잘못 동작합니다.
특히 `Set<Entity>`에 담을 때 프록시와 실 객체가 다른 버킷에 들어가는 버그가 생깁니다.

### FetchType.LAZY 원칙

연관 엔티티는 모두 `FetchType.LAZY`로 선언했습니다.
필요할 때만 추가 쿼리를 발생시켜 불필요한 JOIN을 방지합니다.

**예외 — FetchType.EAGER:**
`UserMathProgress.completedCycles`는 `@ElementCollection`으로,
진행률 조회 시 항상 함께 필요하기 때문에 `EAGER`를 예외적으로 사용했습니다.

### 도메인 메서드 캡슐화

엔티티 필드를 외부에서 직접 수정하지 않고, 의미 있는 메서드로 변경합니다.

```kotlin
// LearningAttempt
fun markWrong() {
    attempts++
    updatedAt = LocalDateTime.now()
}

fun markSolved(hintsUsed: Int) {
    this.solved = true
    this.hintsUsed = hintsUsed
    this.solvedAt = LocalDateTime.now()
    this.updatedAt = LocalDateTime.now()
}
```

"오답 처리"와 "정답 처리"라는 도메인 개념이 메서드 이름으로 드러납니다.

## 3. 전역 예외 처리

```kotlin
@RestControllerAdvice
class GlobalExceptionHandler {
    @ExceptionHandler(IllegalArgumentException::class)
    fun handleIllegalArgument(...) → 400 BAD_REQUEST

    @ExceptionHandler(IllegalStateException::class)
    fun handleIllegalState(...)    → 400 INVALID_STATE

    @ExceptionHandler(NoSuchElementException::class)
    fun handleNotFound(...)        → 404 NOT_FOUND
}
```

도메인 로직의 `require()` / `check()` / `orElseThrow { NoSuchElementException() }`가
자동으로 적절한 HTTP 상태코드로 매핑됩니다.
컨트롤러에서 try-catch를 반복하지 않고 도메인 예외가 그대로 HTTP 응답으로 이어집니다.

```kotlin
// 서비스 레이어
check(todo.member.id == memberId) { "본인의 할 일만 완료할 수 있습니다." }
// → GlobalExceptionHandler → 400 { "code": "INVALID_STATE", "message": "..." }
```

## 4. Kotlin + Spring Boot 선택

### Kotlin을 선택한 이유

| 항목 | Java | Kotlin |
|------|------|--------|
| Null 안전성 | Optional / `@Nullable` | 컴파일 타임 null 검사 (`String?` vs `String`) |
| 데이터 클래스 | record (제한적) | `data class` — copy, destructuring 포함 |
| 코드량 | 보일러플레이트 많음 | 간결한 표현식 |

`Member.grade: Int?`, `Member.nickname: String?`처럼 온보딩 전후 상태를 타입으로 표현하면
서비스 레이어에서 null 체크 누락을 컴파일 시점에 잡을 수 있습니다.

### Spring Boot 3.4를 선택한 이유

- Jakarta EE 9+로 전환 — JDK 21과 함께 장기 지원 경로
- `@Transactional` + `TransactionalEventListener`로 이벤트·트랜잭션 선언적 제어
- springdoc-openapi 2.x Swagger UI 기본 통합
