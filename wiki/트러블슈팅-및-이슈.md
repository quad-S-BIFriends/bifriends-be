## 상세

### Firestore DB ID (#8)

**증상:** 지난 마음 여행·친구랑 503, `(default) does not exist` 로그.

**원인:** 콘솔은 named DB `bifriends`인데 BE는 `(default)` 연결.

**조치:** `application.yml`에 `firebase.firestore.database-id`, deploy `.env` / compose 동기화.

**검증:** `[Firestore] named database 사용 — databaseId=bifriends`

---

### 채팅 트랜잭션 (#5, #7)

**증상:** TX 분리 후 톡톡 전체 장애(read-only TX에서 INSERT 실패) 또는 동시 채팅 시 다른 API 지연.

**조치:**
- `ChatMessageWriteService` 분리
- `sendMessage` → `Propagation.NOT_SUPPORTED`
- 클래스 기본 `readOnly=true`와 INSERT 충돌 해소

---

### `todosCreated` (#6)

**증상:** Leo 할 일 등록 후 채팅 500.

**원인:** AI는 객체 배열, BE는 `long[]` 기대.

**조치:** `ChatTodoCreated(title, assigned_date)` DTO 통일.

---

### 리포트 N+1 (#3)

**증상:** 리포트 목록 조회 시 주차마다 safety SELECT.

**조치:** `weekly_safety_report` 일괄 조회 후 `associateBy weekStart`.

---

### learnedExpressions 스캔 (#4, #9)

**증상:** 시나리오 생성마다 `mindSessions` 전체 읽기.

**조치:** `users/{memberId}.learnedExpressions` 배열 + 저장 시 `arrayUnion`, 레거시 backfill.

---

### 보호자 리포트 (#1)

**증상:** 초기 리포트 생성·조회 오류.

**조치:** 리포트 콜백·조회 플로우 정리 (이후 비동기 타이밍 이슈는 운영 가이드 참고).

**참고:** 리포트 생성은 AI 비동기 — 생성 직후 목록만 있고 `sections` 미완일 수 있음. 폴링 또는 재조회 필요.

## 6. Chat Hikari 커넥션 풀 부하 테스트

📄 **상세 문서:** [doc/chat/hikari_pool_test.md](https://github.com/quad-S-BIFriends/bifriends-be/blob/main/doc/chat/hikari_pool_test.md)

### 배경 (STAR — Situation / Task)

- **Leo 채팅(톡톡)** 사용 시·동시 API 사용 시 체감 지연
- 코드 분석 결과: AI 대기 중 DB 커넥션 점유
- TX 분리 후 **정량 검증** 필요

### 조치 (Action)

- `ChatHikariPoolComparisonTest` — **Leo 채팅 `ChatService.sendMessage()`** 12건 동시 호출
- LEGACY(단일 TX+sleep) vs FIXED(현재 TX 분리) 비교
- H2, pool=5, AI mock 2초 (실제 bifriends-ai HTTP 아님)

### 결과 (Result)

| 지표 | LEGACY | FIXED |
|------|--------|-------|
| max threadsAwaiting | **8** | **0** |
| **채팅 12건 처리 총 소요** | ~6.2s | ~2.0s |
| DB 프로브 max | ~4.2s | 0ms |

**해석:** Leo 응답 시간 자체보다, **DB 풀 고갈로 다른 API가 막히지 않게** 한 개선.

### 실행

```powershell
$env:JAVA_HOME = "C:\Program Files\Java\jdk-17"
.\gradlew test --tests "com.bifriends.domain.chat.ChatHikariPoolComparisonTest"
```
