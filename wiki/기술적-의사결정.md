# 5. 기술적 의사결정

주요 선택과 **왜 그렇게 했는지**만 정리합니다. 상세는 레포 `doc/` 참고.

---

## 인증: Firebase + JWT

| 선택 | 이유 |
|------|------|
| 로그인은 Firebase Google OAuth | 클라이언트 SDK·검증 성숙 |
| API는 자체 JWT | 매 요청 Firebase 검증 비용·지연 감소, Stateless |

흐름: `ID Token` → `POST /members/auth/google` → `accessToken` → 이후 `Bearer` 헤더

---

## DB 이원화: PostgreSQL + Firestore

| PostgreSQL | Firestore |
|------------|-----------|
| 채팅, 학습, 리포트, 상점 | 친구랑 세션(완료분만) |
| 관계·집계·트랜잭션 | JSON-heavy, 이미지 URL |

친구랑은 **시나리오 생성과 완료 저장 분리** (`/mind/scenario` vs `/mind/sessions`) — 중간 이탈 시 DB 오염 방지.

---

## 채팅: 트랜잭션과 AI 호출 분리

**문제:** `@Transactional` 안에서 AI HTTP 대기 → Hikari 커넥션 장시간 점유.

**결정:**
1. `ChatMessageWriteService` — DB만 짧은 TX
2. `ChatService.sendMessage` — `@Transactional(NOT_SUPPORTED)` 로 AI 구간 TX 없음

**트레이드오프:** AI 실패 시 user 메시지는 DB에 남을 수 있음 (UX상 자연스러움).

→ 정량 검증: [[트러블슈팅-및-이슈#6-chat-hikari-커넥션-풀-부하-테스트]]

---

## AI 연동: 동기 중계 vs 비동기 배치

| 패턴 | 예 |
|------|-----|
| 동기 | 채팅 `POST /chat/messages` |
| 스케줄러 → AI → 콜백 | 주간 리포트, 안전 신호 |

배치 실패는 `AiBatchClient`에서 로그만 — 앱 핵심 경로 차단 안 함.
