# 서비스 간 통신 아키텍처

> **3개 레포** (`bifriends-client`, `bifriends-be`, `bifriends-ai`)는 **코드를 공유하지 않는다.**  
> 통신은 **HTTP REST + 환경변수 URL + API 명세 문서**로만 맞춘다. BE가 **유일한 중계자(BFF)** 이다.

| 레포 | 스택 | 역할 |
|------|------|------|
| `bifriends-client` | Flutter | 앱 UI |
| `bifriends-be` | Kotlin / Spring Boot | API·DB·인증·AI 중계 |
| `bifriends-ai` | Python / FastAPI | Leo·시나리오·주간 배치 AI |

[[기술적-의사결정]] · [[인증-JWT-및-Internal-Token]] · [`doc/presentation/fe_ai_communication.md`](https://github.com/quad-S-BIFriends/bifriends-be/blob/test/chat/doc/presentation/fe_ai_communication.md)

---

## 1. 기술적 결정 요약

| 결정 | 이유 |
|------|------|
| **FE ↔ AI 직접 통신 금지** | API 키·스펙·보안을 BE 한곳에서 통제 |
| **BE ↔ AI = HTTP REST** | 언어·레포 분리 유지, 팀 독립 배포 |
| **코드 공유 없음** | `doc/ai/api_spec.md` 등 **계약 문서**로 DTO·경로 정합 |
| **배포 = Docker Compose** | BE·AI·DB를 **같은 내부 네트워크**에 올려 서비스명으로 호출 |
| **외부 노출은 BE만** | Nginx → `api.bifriends.study`, AI 포트는 내부 전용 |

인증 상세: **[[인증-JWT-및-Internal-Token]]**

## 2. 전체 통신 다이어그램

```
bifriends-client (Flutter)     bifriends-be (Kotlin)        bifriends-ai (Python)
        │                              │                            │
        │  HTTPS + JWT                 │                            │
        └─────────────────────────────▶│  HTTP (Docker 내부)         │
                                       └───────────────────────────▶│
                                       ◀───────────────────────────┘
                                         X-Internal-Service (콜백)
```

| 화살표 | 프로토콜 | 인증 | 설명 |
|--------|----------|------|------|
| FE → BE | **HTTPS** | `Bearer JWT` | 앱이 호출하는 유일한 백엔드 |
| BE → AI | HTTP (내부) | 없음 | `RestClient` + `AI_SERVICE_BASE_URL` |
| AI → BE | HTTP (내부) | `X-Internal-Service` | 콜백·할 일·이력 조회 등 |
| FE → AI | — | **없음** | 설계상 금지 |

## 3. 레포가 달라도 통신하는 방법

### 공유하지 않는 것

- Gradle/pip 패키지 의존
- 공통 라이브러리 JAR
- Git submodule

### 맞추는 것

| 항목 | 방법 |
|------|------|
| URL | 환경변수 (`AI_SERVICE_BASE_URL`, `BE_BASE_URL`) |
| JSON 스키마 | BE `infrastructure/ai/dto`, AI Pydantic 모델 — **문서로 동기화** |
| 내부 API | `doc/ai/api_spec.md` (BE → AI 팀) |
| 필드명 | FE camelCase ↔ BE camelCase ↔ AI snake_case (BE에서 변환) |

**예:** `todosCreated` 스펙 불일치 → [[todosCreated-스펙-통일]]

## 4. BE → AI (호출)

### 구현

- `AiChatClient`, `AiBatchClient`, `AiEmotionScenarioClient`
- Spring `RestClient` — `AiClientConfig`에서 `baseUrl`·타임아웃(연결 15s, 읽기 3분) 설정

```kotlin
// AiChatClient.kt — 요약
restClient.post()
    .uri(properties.chatPath)   // 예: /api/v1/ai/chat
    .postJson(request)
    .retrieve()
    .body(AiChatResponse::class.java)
```

### 환경변수

```yaml
ai:
  service:
    enabled: ${AI_SERVICE_ENABLED:false}
    base-url: ${AI_SERVICE_BASE_URL:http://bifriends-ai:8001}
    chat-path: ${AI_CHAT_PATH:/api/v1/ai/chat}
```

| 호출 종류 | 예시 경로 | 패턴 |
|-----------|-----------|------|
| Leo 채팅 | `POST /api/v1/ai/chat` | 동기 (FE 요청 안에서) |
| 감정 시나리오 | `POST /api/v1/ai/content/scenario` | 동기 |
| 주간 배치 | `POST /api/v1/ai/batch/...` | 스케줄러 → AI → AI가 BE 콜백 |

`AI_SERVICE_ENABLED=false` → AI HTTP 생략 (로컬 BE 단독 개발)

## 5. AI → BE (콜백·내부 API)

AI는 `BE_BASE_URL` + 화이트리스트 경로만 호출한다.

| 용도 | 메서드·경로 |
|------|-------------|
| Leo 할 일 | `POST/PATCH/DELETE /api/v1/todos` |
| 채팅 이력 | `GET /api/v1/chat/messages` |
| 주간 리포트·안전 | `POST /api/v1/weekly-report`, `weekly-safety-report` |
| 학습 요약 | `GET /api/v1/report/learning-summary` |

헤더: `X-Internal-Service: {INTERNAL_SERVICE_TOKEN}`  
화이트리스트: `InternalServicePaths.kt`

## 6. 배포 환경 (GCP VM)

### 디렉터리

```
/app/
├── bifriends-be/     ← git clone
│   └── deploy/
│       ├── docker-compose.yml
│       └── .env
└── bifriends-ai/     ← git clone (형제 폴더)
```

### docker-compose 서비스

| 서비스 | Docker 이름 | URL (컨테이너 간) | 외부 노출 |
|--------|-------------|-------------------|-----------|
| DB | `bifriends-db` | `db:5432` | ❌ |
| BE | `bifriends-be` | `bifriends-be:8080` | ✅ 호스트 8080 |
| AI | `bifriends-ai` | `bifriends-ai:8001` | ❌ |

### BE 컨테이너 env (발췌)

```yaml
AI_SERVICE_ENABLED: "true"
AI_SERVICE_BASE_URL: http://bifriends-ai:8001
INTERNAL_SERVICE_TOKEN: ${INTERNAL_SERVICE_TOKEN}
```

### AI 컨테이너 env (발췌)

```yaml
BE_BASE_URL: http://bifriends-be:8080
INTERNAL_SERVICE_TOKEN: ${INTERNAL_SERVICE_TOKEN}
GOOGLE_API_KEY: ${GOOGLE_API_KEY}
```

`bifriends-ai` build context: `../../bifriends-ai` — **별도 레포를 compose가 빌드**

### 외부 트래픽

```
Flutter  ──HTTPS──▶  Nginx (443)  ──▶  localhost:8080 (BE)
api.bifriends.study
```

- `deploy/nginx.conf` — **BE만** 프록시
- AI는 인터넷에 포트 미개방

### 배포·갱신

- 수동: `cd /app/bifriends-be/deploy && docker compose build && docker compose up -d`
- CI: `main` push → VM SSH → `git pull` + BE 이미지 재빌드 (AI 변경 시 AI pull·재빌드 별도)

상세: [`deploy/DEPLOY.md`](https://github.com/quad-S-BIFriends/bifriends-be/blob/test/chat/deploy/DEPLOY.md)

## 7. 개발 환경

개발은 **한 가지 방식만 쓰지 않는다.**

### 패턴 A — BE 단독 (가장 흔함)

```bash
docker-compose up -d db    # Postgres만
./gradlew bootRun          # :8080
```

| 항목 | 값 |
|------|-----|
| `AI_SERVICE_ENABLED` | 기본 **`false`** |
| BE → AI | 호출 안 함 (`reply: null`) |

### 패턴 B — BE + AI 로컬 연동

- `bifriends-ai` 레포에서 AI 서버 실행 (예: `:8001`)
- BE env:

```bash
AI_SERVICE_ENABLED=true
AI_SERVICE_BASE_URL=http://localhost:8001
```

- AI env: `BE_BASE_URL=http://localhost:8080` (또는 Docker AI면 `host.docker.internal`)

루트 `docker-compose.yml`에는 **AI 서비스 없음** — BE+DB만.

### 패턴 C — 배포와 동일 스택

```bash
cd deploy
docker compose up -d    # BE + AI + DB 전부
```

Docker DNS `bifriends-ai`, `bifriends-be` 그대로 사용.

## 8. 개발 vs 배포 비교

| | **개발 (일반)** | **배포 (VM)** |
|---|----------------|---------------|
| 레포 | 각자 clone | `/app/` 형제 폴더 |
| BE 실행 | `bootRun` 또는 로컬 Docker | `deploy/docker-compose` |
| AI | often OFF 또는 별도 실행 | compose에 **항상 포함** |
| BE→AI URL | `localhost:8001` (켤 때) | `http://bifriends-ai:8001` |
| AI→BE URL | `localhost:8080` | `http://bifriends-be:8080` |
| Internal token | 기본값 또는 `.env` | `openssl rand -hex 32` |
| 앱이 보는 URL | `localhost:8080` | `https://api.bifriends.study` |

## 9. 동기 vs 비동기 흐름

### 동기 — Leo 채팅

```
FE ─JWT─▶ BE ─HTTP─▶ AI
FE ◀────── BE ◀────── AI
```

사용자가 응답을 기다리는 경로. BE가 `ChatService`에서 AI 호출 후 DB 저장.

### 비동기 — 주간 리포트·안전 배치

```
BE Scheduler ─HTTP─▶ AI (배치 시작)
AI ─X-Internal─▶ BE (콜백으로 결과 저장)
FE ─JWT─▶ BE (나중에 조회)
```

AI 처리 시간이 길어도 앱 핵심 API를 막지 않음.

## 면접 한 줄

> “레포는 Flutter·Kotlin·Python으로 나뉘지만, **BE를 BFF로 두고 HTTP만** 씁니다. 배포는 VM에서 docker-compose로 BE·AI를 같은 네트워크에 올리고 Docker 서비스명으로 호출하며, 외부에는 Nginx로 BE만 노출합니다. 개발은 DB만 Docker + bootRun이 기본이고 AI는 env로 켜고 끕니다.”
