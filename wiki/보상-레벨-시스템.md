# 보상·레벨 시스템 설계

[[기술적-의사결정]]

---

## 핵심 설계 원칙

```
totalPoolEarned  → 누적 획득 (절대 감소 안 함) → 레벨 계산 기준
availablePool    → 현재 보유 (상점 소비 시 감소) → 구매 가능 여부
```

레벨은 **학습 진행의 기록**이므로 상점에서 포인트를 써도 레벨이 내려가서는 안 됩니다.
두 필드를 분리함으로써 서비스 레이어에서 실수할 여지를 없앴습니다.

---

## Policy Object 패턴

비즈니스 규칙이 서비스·엔티티 곳곳에 흩어지면 규칙 변경 시 수정 포인트가 많아집니다.
Kotlin `object`로 정책을 격리하여 **변경 시 한 곳만 수정**하도록 했습니다.

### RewardPolicy

```kotlin
object RewardPolicy {
    // 출석 연속 일수에 따른 차등 보상
    fun attendancePool(streakDays: Int): Int = when {
        streakDays >= 3 -> 5
        streakDays == 2 -> 4
        else            -> 3  // 첫 로그인 / 재방문
    }

    const val LEARNING_CORRECT      = 1   // 문제 1개 정답
    const val LEARNING_SET_COMPLETE = 2   // 3문제 세트 완료 보너스
    const val EMOTION               = 3   // 친구랑 시나리오 완료
    const val TODO_SINGLE           = 1   // 할 일 단건 완료
    const val TODO_ALL_COMPLETE     = 3   // 할 일 전체 완료 보너스
}
```

### LevelPolicy

```kotlin
object LevelPolicy {
    // Lv N → N+1 필요 풀 = 10 + (N-1) × 5
    // Lv1→2: 10풀, Lv2→3: 15풀, Lv3→4: 20풀, ...
    fun poolRequiredForLevelUp(fromLevel: Int): Int = 10 + (fromLevel - 1) * 5

    // 레벨 N 도달까지 필요한 누적 풀 = 5 × (N-1) × (N+2) / 2
    fun cumulativePoolForLevel(level: Int): Int { ... }

    fun calculateLevel(totalPoolEarned: Int): Int { ... }
}
```

`LevelPolicy`는 순수 함수로만 구성되어 있어 단위 테스트가 쉽습니다.

## 엔티티 도메인 메서드

풀 지급·소비 로직은 `UserStats` 엔티티 내부에 캡슐화했습니다.
외부에서 필드를 직접 수정하는 것을 방지합니다.

```kotlin
// 풀 획득 — totalPoolEarned, availablePool, level을 한 번에 갱신
fun earnPool(amount: Int): Boolean {
    require(amount > 0) { "획득 풀은 1 이상이어야 합니다." }
    totalPoolEarned += amount
    availablePool += amount
    val newLevel = LevelPolicy.calculateLevel(totalPoolEarned)
    val leveledUp = newLevel > level
    level = newLevel
    updatedAt = LocalDateTime.now()
    return leveledUp
}

// 풀 소비 — availablePool만 감소, 레벨 불변
fun spendPool(amount: Int) {
    require(amount > 0) { "소비 풀은 1 이상이어야 합니다." }
    check(availablePool >= amount) { "풀이 부족합니다." }
    availablePool -= amount
    updatedAt = LocalDateTime.now()
}
```

## 보상 지급 흐름 (예: 할 일 완료)

```kotlin
// TodoService.completeTodo
todo.complete()  // 엔티티 도메인 메서드 → 이미 완료면 IllegalStateException

// 단일 완료 보상
userStatsService.earnReward(
    memberId = memberId,
    source = RewardSource.TODO_SINGLE,
    amount = RewardPolicy.TODO_SINGLE,
    refId = todoId,
)

// 전체 완료 보너스 체크
val remainingPending = todoRepository.countByMemberIdAndAssignedDateAndStatus(...)
if (remainingPending == 0) {
    userStatsService.earnReward(source = RewardSource.TODO_ALL_COMPLETE, ...)
}
```

모든 처리는 단일 트랜잭션 — `UserStats` 갱신과 `RewardHistory` 삽입이 원자적으로 처리됩니다.

## 보상 이력 (`reward_history`)

```sql
CREATE TABLE reward_history (
    id         BIGSERIAL PRIMARY KEY,
    member_id  BIGINT      NOT NULL,
    source     VARCHAR(30) NOT NULL,  -- TODO_SINGLE, LEARNING_CORRECT 등
    amount     INTEGER     NOT NULL,
    ref_id     BIGINT,                -- 참조 ID (todo_id, attempt_id 등)
    created_at TIMESTAMP   NOT NULL
);
```

모든 보상 지급을 이력으로 남겨 디버깅과 추후 분석에 활용할 수 있습니다.
