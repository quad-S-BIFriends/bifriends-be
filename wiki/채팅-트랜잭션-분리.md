## STAR

### Situation

**#5 — 성능:** `@Transactional` 안에서 AI HTTP(수 초~수 분) 대기 → Hikari 커넥션 장시간 점유 → 동시 사용 시 다른 API 지연.

**#7 — 회귀:** TX 분리 과정에서 `ChatService`가 클래스 단위 `readOnly=true`인데 INSERT 시도 → 톡톡 **전면 장애**.

### Task

- AI 대기 구간에 DB 커넥션을 붙잡지 않기
- user/assistant 메시지 INSERT는 정상 동작

### Action

1. `ChatMessageWriteService` — DB 저장만 담당하는 짧은 `@Transactional`
2. `ChatService.sendMessage` — `@Transactional(propagation = NOT_SUPPORTED)` 로 AI 구간 TX 없음
3. read-only 회귀: WriteService로 INSERT 분리, `NOT_SUPPORTED`로 상위 readOnly TX 탈출

### Result

- 톡톡 저장 정상 복구 (#7)
- 부하 테스트: `threadsAwaiting` 8 → 0, 채팅 12건 ~6.2s → ~2.0s → [[Chat-Hikari-풀-테스트]]

## 코드 요약

```kotlin
@Transactional(propagation = Propagation.NOT_SUPPORTED)
fun sendMessage(...) {
    chatMessageWriteService.saveUserMessage(...)
    val aiResponse = aiChatClient.sendChat(...)
    aiResponse.reply?.let { chatMessageWriteService.saveAssistantMessage(...) }
}
```

**트레이드오프:** AI 실패 시 user 메시지는 DB에 남을 수 있음 (UX상 자연스러움).
